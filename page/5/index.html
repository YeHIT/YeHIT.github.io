Dubbo负载均衡及其他常见的负载均衡方法

之前主要详细分析了Dubbo内部服务暴露的过程，可以见Dubbo服务暴露流程 | Yesomething，在分析过程中发现Dubbo服务引用的过程本质上和服务暴露过程区别不大。而Dubbo调用服务的过程调用链过长，且使用了太多自定义SPI动态加载类和代理模式，这里就不具体展开源码了。其核心流程大致可以分为：(预先已经完成服务引用) 构造请求 -> 序列化 -> 服务提供者反序列化 -> 根据序列化结果触发对应方法 -> 方法返回结果封装 -> 序列化发送给调用方 -> 调用方解析结果 -> 结束调用。

但是这调用链的过程中，发现Dubbo提供了许多有趣的组件，如可以用于限流/服务降级MockInvoker、Dubbo服务失败failover机制、Dubbo调用服务的负载均衡方式等。

这篇文章主要分析Dubbo提供的负载均衡方式，并尝试了解更多常见的负载均衡方法。

Dubbo内置的负载均衡策略

random loadbalance

按权重设置随机概率。然后权重不同会让每个服务提供者占中权重的一个小区间，通过随机数产生对应offset，落到哪个区间就使用哪个服务提供者。

如：服务A的三个提供者，权重分别为100，50，200，记为A1、A2、A3。则A1占有[0，100)区间，A2占有[100，150)区间，A3占有[150,350)区间。然后通过生成[0,350)范围内的随机数来确定使用哪个服务提供者。

这个方式基本随机，可能随机出分布不均匀的情况。

roundrobin loadbalance

轮询，按公约后的权重设置轮询比例。

该类内部维护了一个以<服务名为key，对应服务提供者作为Map>的ConcurrentMap。具体实现为ConcurrentMap<String, ConcurrentMap<String, WeightedRoundRobin>>。

假定服务A的三个提供者，权重分别为100，50，200，记为A1、A2、A3。

类似随机负载均衡，但是过程更加合适，分布均匀。

阶段 A1 A2 A3 TotalWeight
初始化 0 0 0 0
第一次选用 0+100 = 100 0 + 50 = 50 0 + 200 = 200 100 + 50 + 200 = 350
选用A3完成 100 50 200- 350 = -150 0
第二次选用初始化 100 50 -150 0
第二次选用 100 + 100 = 200 50 + 50 = 100 -150 + 200 = 50 350
选用A1完成 200 - 350 = -150 100 50 0

leastactive loadbalance

最少活跃调用数，相同活跃数的随机，活跃数指调用前后计数差。使慢的提供者收到更少请求。

所谓活跃数，可以理解为当前服务提供者正在给多少消费者提供服务。

这个负载均衡策略即为先找出活跃数最少的那批服务提供者，然后根据服务提供者的权重再按照随机负载均衡选取。

consistanthash loadbalance

一致性Hash，相同参数的请求总是发给同一个提供者。默认一致性hash使用160个虚拟节点。

ShortestResponseLoadBalance

最短响应时间负载均衡。Dubbo的响应时间为服务提供者响应服务的平均时间 * 当前服务提供者的活跃数。

代码流程即：先找出响应时间最短的那批服务提供者，找出后再通过随机负载均衡来选择服务提供者。
